# Phase 3: Rich Index Page - Context

**Gathered:** 2026-04-06 (assumptions mode, --auto)
**Status:** Ready for planning

<domain>
## Phase Boundary

Generate an `index.html` at the root of the gh-pages branch on every deployment. The page renders the contents of `versions.json` (versions list + per-version commits) into a clean, readable, mobile-friendly timeline. The widget that gets injected into individual deployed pages is Phase 4 — out of scope here.
</domain>

<decisions>
## Implementation Decisions

### Generation strategy
- **D-01:** Server-side render at deploy time, NOT client-side fetch. The action writes a fully-formed `index.html` (with all version data inlined) into the gh-pages worktree on every deploy. Reason: works without JS, no extra network round-trip, simpler hosting story (gh-pages serves a static file), and the manifest already lives at a known path.
- **D-02:** New module `src/index-renderer.ts` exports `renderIndexHtml(manifest: Manifest, repoMeta: { owner: string; repo: string }): string`. Pure function — takes the manifest, returns HTML string. Zero I/O.
- **D-03:** Wire into pipeline AFTER `writeManifest` and BEFORE the commit step in `branch-manager`. Same atomic commit as the manifest update (MNFST-04 / INDX-06).

### Rendering approach
- **D-04:** Single self-contained HTML file: inline `<style>`, inline `<script>` (only for collapsible commit history details — `<details>` element so it works without JS), no external assets. Reason: zero asset hosting concerns, no CORS, no CDN dependency, deploys atomically.
- **D-05:** No template engine. Build the HTML by string concatenation in TypeScript with a small `escapeHtml()` helper. Reason: avoiding any new dependency keeps the bundle small and the action portable. The output is small enough that template engines are overkill.
- **D-06:** ALL user-controlled strings (version names, refs, commit messages, author names/emails) go through `escapeHtml()`. Threat: stored XSS via crafted commit messages or branch names that land in the rendered index. Mitigation: single escape helper applied at every interpolation point — enforced by code review (no raw `${...}` of user data permitted).

### Visual design
- **D-07:** **Layout:** Vertical timeline. Each version is a card containing version name (heading), git ref + commit SHA + deploy timestamp (subtitle row), and a `<details>` element labeled "N commits" that expands to a list of commits. Most-recent version at the top.
- **D-08:** **Colors:** Light theme with `prefers-color-scheme: dark` override. Neutral palette (slate/gray text on white/near-black background, single accent color for links and active state). Specific tokens locked in UI-SPEC.md.
- **D-09:** **Typography:** System font stack — `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`. No web fonts. Mono font (`ui-monospace, "SF Mono", Menlo, monospace`) for SHAs only.
- **D-10:** **Layout primitives:** CSS only (flex + grid), no framework. `max-width: 760px` content column centered. Mobile breakpoint at 600px collapses card padding and shrinks heading sizes.
- **D-11:** **Header:** Repo name (`{owner}/{repo}`) as h1, link back to GitHub repo URL, "{N} versions deployed" subtitle. Footer is a single line: "Generated by gh-pages-multiplexer at {ISO timestamp}" — verifiable freshness signal.

### Empty / edge cases
- **D-12:** **Empty manifest** (zero versions) renders a placeholder card: "No versions deployed yet." This case shouldn't happen in practice (the index is rendered after a successful deploy that adds at least one version) but the renderer must not crash on it.
- **D-13:** **Missing commits[]** (legacy schema-1 entries from before Phase 2) render the version card without the commits `<details>` element — no error, no empty section. Per-entry presence is data-driven, not a control-flow gate.
- **D-14:** **Long commit messages** are rendered with the full subject line; the body is clipped after the first line (`message.split('\n')[0]`). Full body is preserved in the manifest so future enhancements can reveal it.

### Out of scope (this phase)
- **D-15:** Per-version preview thumbnails, search/filter UI, version sorting controls, version diff view — all out of scope. Defer to v2.
- **D-16:** Author avatars (would require gravatar lookup or GitHub API) — defer; the index shows author name + email only.

### Claude's Discretion
- Exact spacing values and corner radii (UI-SPEC.md will lock these, but planner has latitude inside the spec).
- Whether to gzip/minify the inline HTML (not worth the complexity for ~5KB).
- Exact accent color hex (locked in UI-SPEC.md).
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `Manifest` and `CommitInfo` from Phase 2 already carry exactly the fields the index needs to render. No new data extraction required.
- `branch-manager.ts` already has the gh-pages worktree open with write access at the relevant point in the pipeline — adding one more file (`index.html`) costs zero extra I/O machinery.

### Established Patterns
- Pure data transforms in modules; single I/O enforcer for gh-pages writes lives in `branch-manager.ts`. The renderer is a pure function that produces a string; the existing single-enforcer writes it to disk. No violation.
- `[LAW:dataflow-not-control-flow]` annotation on `src/index.ts`: render is unconditional (no "if has commits" branch in the pipeline; the renderer handles empty/missing data internally via D-12 / D-13).
- Tests build real fixture data (no mocks) and assert observable behavior (DOM structure or rendered substrings, not internal calls).

### Integration Points
- Hook 1: New `src/index-renderer.ts` file
- Hook 2: `src/branch-manager.ts` — after `writeManifest`, call `renderIndexHtml(manifest, repoMeta)` and write the result to `<workdir>/index.html`. Stage both files before the commit.
- Hook 3: `src/index.ts` — no changes; `branch-manager` owns the rendering call as part of its commit responsibilities.
- Hook 4: Build a `__tests__/index-renderer.test.ts` exercising structure (heading present, version count correct, escapes work, empty manifest, legacy entry without commits) and a smoke test that pipes through the live pipeline test.

</code_context>

<specifics>
## Specific Ideas

- The `<details>` element for commit history gives free zero-JS expand/collapse — works on every browser, zero dependency, accessible by default.
- Inlining everything makes the index file ~5–20KB depending on commit count — well under any practical limit and trivially cacheable.
- The "Generated at {timestamp}" footer is a deliberate freshness/observability signal: one glance at the deployed index tells you when the action last ran.

</specifics>
