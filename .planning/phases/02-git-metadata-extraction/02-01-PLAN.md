**Contract:**
```typescript
export async function extractCommits(
  repoDir: string,
  currentSha: string,
  previousSha: string | null,
): Promise<CommitInfo[]>
```

**Law annotations:**
- `// [LAW:dataflow-not-control-flow]` — this module is the one exception where we shell out; keep the structure a single unconditional git-log invocation whose *arguments* vary, not whose code path varies.
- `// [LAW:one-source-of-truth]` META-03 — the manifest is the sole home for this data; this module is the producer.
- `// [LAW:single-enforcer]` — only this module invokes git log.

**Implementation outline:**

1. Import `execFile` from `node:child_process`, promisify it. Import `* as core from '@actions/core'`. Import `CommitInfo` from `./types.js`.

2. Define constants:
   ```typescript
   const MAX_COMMITS = 100;
   const FIELD_SEP = '\x1f';
   const RECORD_SEP = '\x1e';
   const FORMAT = `%H${FIELD_SEP}%an${FIELD_SEP}%ae${FIELD_SEP}%aI${FIELD_SEP}%B${RECORD_SEP}`;
   ```

3. Helper `runGitLog(repoDir: string, revRange: string): Promise<string>` — invokes
   `git log --format=<FORMAT> -n <MAX_COMMITS> <revRange>` via execFile in `repoDir`.
   Do NOT swallow errors. On rejection, inspect stderr; if it mentions "unknown revision" or "bad revision" (force-push case), throw a tagged error the caller can catch. For any other failure, wrap the stderr in an Error whose message includes: `git log failed in ${repoDir}: ${stderr}. If this is a shallow clone, set fetch-depth: 0 on actions/checkout.` Always include the fetch-depth hint on generic failure per D-11.

4. Helper `parseLog(stdout: string): CommitInfo[]` — split on RECORD_SEP, drop the trailing empty chunk, for each record split on FIELD_SEP into `[sha, author_name, author_email, timestamp, message]`. Message is whatever remains after the 4th FIELD_SEP (do not trim; preserve newlines). Return the array in the order git emits (newest first).

5. Main function logic:
   - Compute `revRange`: if `previousSha === null`, use `currentSha`. Otherwise, use `${previousSha}..${currentSha}`.
   - Call `runGitLog(repoDir, revRange)`.
   - On the "unreachable previous SHA" tagged error, `core.info(\`Previous SHA ${previousSha} not reachable (force-push?); falling back to full history capped at ${MAX_COMMITS}.\`)` and retry with `revRange = currentSha`. Any other error rethrows.
   - Parse the output. Slice defensively to at most `MAX_COMMITS` (git's `-n` already enforces this, but belt-and-suspenders for the case where the format yields unexpected records).
   - If the result length equals `MAX_COMMITS` and there were likely more commits, emit `core.info(\`Commit history capped at ${MAX_COMMITS} for this deployment.\`)`. For the incremental range case, checking "likely more" is hard; emit the cap notice whenever the slice truncated or whenever the slice hit MAX_COMMITS (acceptable per D-06: "silently dropped with a core.info line").
   - Return `CommitInfo[]`.

6. No null guards beyond what the spec requires. No `|| []` fallbacks. Let unexpected errors propagate.

Run `npx vitest run __tests__/metadata-extractor.test.ts` — all 7 tests must pass. Run `npx tsc --noEmit` — clean. Commit: `feat(02-01): implement metadata-extractor with git log shell-out`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source repo -> extractCommits | Commit data (author name, email, message) may contain attacker-authored content that flows into the manifest and downstream renderers |
| execFile arguments | SHAs passed as revRange arguments must not be interpreted as shell syntax |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Injection | runGitLog revRange argument | mitigate | Use `execFile` (not `exec`/shell) so `previousSha..currentSha` is a single argv element. No shell interpolation. SHAs originate from the manifest (internally written) and the deployment context (GitHub-provided). |
| T-02-02 | Tampering | commit message content stored in manifest | accept | Commit messages are preserved verbatim; downstream renderers (Phase 3 index page) are responsible for HTML-escaping. Storing them raw is the correct layering. |
| T-02-03 | DoS | pathological commit history size | mitigate | Hard cap of 100 commits (D-06) bounds manifest growth regardless of input. |
| T-02-04 | Information Disclosure | author emails in public manifest | accept | Author emails are already public via git history on the source repo; copying them to the manifest does not expand the disclosure surface. |
</threat_model>
